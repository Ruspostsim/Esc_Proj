# Веб-камера → Media Texture (UE 5.7, Win64, PIE и Shipping)

Руководство для проекта **WILL_DESIRE_Anim**: вывод живого видео с ПК-камеры в игру через **Media Framework** и плеер **WMF** (Windows Media Foundation). Такой путь поддерживается в редакторе, в Standalone Game и в **упакованной** сборке Win64.

Официальная база от Epic: [Playing Live Video Captures in Unreal Engine 5.7](https://dev.epicgames.com/documentation/en-us/unreal-engine/playing-live-video-captures-in-unreal-engine?application_version=5.7).

## Что уже сделано в репозитории

- Плагин **`WebcamMedia`** (`Plugins/WebcamMedia/`) подключён в `WILL_DESIRE_Anim.uproject`.
- Он тянет за собой **`WmfMedia`** только для **Win64** (в `WebcamMedia.uplugin` указан `PlatformAllowList`).
- Класс **`UWebcamMediaFeedComponent`** (компонент на акторе): создаёт `UMediaPlayer` / `UMediaTexture`, если вы их не назначили, перечисляет устройства захвата и вызывает `OpenUrl` + `Play`.

Важно: **Electra** для живого захвата с камеры **не** подходит (см. документацию Epic). Используется обычный Media Player + WMF.

## Быстрый старт (компонент)

1. Откройте проект в UE 5.7, дождитесь компиляции плагина.
2. Выберите любой **Actor** (или Blueprint-актор), **Add Component** → **Webcam Media Feed** (класс `WebcamMediaFeedComponent`).
3. В деталях компонента:
   - **`Device Index`**: `0` — первое устройство из списка перечисления (если камера не первая — увеличьте индекс).
   - **`Video Capture Device Filter`**: `-1` — все типы; **`8`** — только флаг **Webcam** (`EMediaVideoCaptureDeviceFilter::Webcam`), чтобы отсечь часть виртуальных/прочих устройств.
   - **`Custom Capture Url`**: оставьте пустым для автоподбора; либо вставьте URL, скопированный из ассета **Media Player** (см. ниже «ручной URL»).
4. В **UMG** или на меш: назначьте **`Media Texture`** из компонента (если оставили поля пустыми — это будет внутренняя текстура; в Blueprint можно взять ссылку на переменную, если назначите текстуру вручную).
5. Запустите PIE: при **`b Auto Start`** = true поток откроется в `BeginPlay`.

Лог категории **`LogWebcamMedia`**: при старте пишется открытый URL; при ошибке — список найденных устройств.

## Вариант без C++ (чистые ассеты Epic)

Повторяет официальный туториал:

1. **Content Browser** → **Media** → **Media Player** (с галочкой «создать связанную Media Texture»).
2. Откройте **Media Player**, в поле **Media URL** откройте список **Capture Devices**, выберите камеру, убедитесь что превью идёт.
3. Скопируйте строку **Media URL** в буфер.
4. В **Level Blueprint** / своём акторе на **Begin Play**: **`Open Url`** у **Media Player** с этой строкой, затем **`Play`** (при необходимости).
5. Для динамического выбора камеры у пользователя — нода **`Enumerate Video Capture Devices`** и **`Open Url`** по `Url` из структуры устройства (как в документации Epic).

Компонент из плагина делает то же самое в C++, чтобы не дублировать логику в каждом Blueprint.

## Как вывести картинку в мир или UI

- **UMG**: виджет **Image**, в **Brush** → **Image** укажите вашу **Media Texture** (или ту, что создаёт компонент).
- **Material**: параметр типа **Texture Sample** / **Media Texture** на эмиссив или базовый цвет; материал на **Widget**, **User Interface** mesh или постпроцесс — по задаче.

## «Пожать» видео по памяти и CPU

Размер **VRAM** и нагрузка декодера в первую очередь задаются **разрешением и форматом потока**, которые отдаёт драйвер камеры, а не масштабом картинки на экране.

Практичные шаги:

1. **Windows**: «Параметры» → «Bluetooth и устройства» → **Камеры** → выберите камеру → где доступно, задайте более низкое разрешение для приложений (зависит от драйвера/устройства).
2. **`Video Capture Device Filter` = `8`**: в списке останутся веб-камеры; иногда это убирает тяжёлые виртуальные устройства (но виртуальная камера OBS может попасть в другой класс — тогда подберите фильтр или фиксируйте **`Custom Capture Url`**).
3. **Project Settings → Plugins → WMF Media**: опция **Low Latency** уменьшает задержку и буфер, ценой качества/стабильности — иногда полезно для «лёгкого» превью.
4. **UI**: рисуйте превью маленьким — визуально дешевле, но текстура всё равно того размера, что отдаёт захват.
5. Если нужен отдельный **низкоразмерный рендер-таргет** для дальнейшей логики (не только отображение), можно в материале или отдельным проходом даунскейлить в **Render Target 2D** меньшего размера — это уже дополнительная стоимость копирования, зато дальнейшие шейдеры работают на меньшем разрешении.

Полностью «пережать» поток на уровне кодека средствами только Blueprint без нативного кода в Media Framework ограничено: для жёсткого контроля битрейта/разрешения обычно смотрят в сторону отдельного захвата (например, собственный модуль на Media Foundation / DirectShow) — это выходит за рамки текущего плагина.

## Упаковка (Shipping / Development)

- Убедитесь, что в **Packaging → Additional Non-Asset Directories to Package** ничего лишнего не требуется: для веб-камеры отдельные файлы контента не обязательны, если вы не используете только редакторские ассеты.
- Плагин **WebcamMedia** и **WmfMedia** должны оставаться **включёнными** для Win64 (у вас это задано в `.uproject` и в зависимостях плагина).
- На целевой машине должна быть доступна камера (права приложения в Windows обычно запрашиваются при первом обращении к устройству — поведение ОС).

## Диагностика

В `DefaultEngine.ini` при необходимости можно временно включить подробные логи медиа (см. [Media Framework Technical Reference](https://dev.epicgames.com/documentation/en-us/unreal-engine/media-framework-technical-reference-for-unreal-engine?application_version=5.7), раздел Troubleshooting — `LogWmfMedia`, `LogMedia` и т.д.).

## Ограничения

- Ориентир текущей реализации — **Win64 + WmfMedia**. Другие платформы (Android-камера, macOS) используют другие плееры и шаги настройки.
- Захват с нескольких камер одновременно — несколько **Media Player** / несколько компонентов с разными индексами или URL.
